# Documentation Work Plan

Goal: turn the current autogenerated docs into a maintained reference for the public API, with examples, provenance, and enough mathematical context for the Faddeeva and AMOS families.

## Current status

- [x] Phase 1: Coverage audit
- [ ] Phase 2: API reference pages
- [ ] Phase 3: Examples
- [ ] Phase 4: Provenance and math notes
- [ ] Phase 5: Plots and benchmarks
- [ ] Phase 6: Final verification and release

## Phase 1: Coverage audit

1. Inventory the exported surface from `src/OpenSpecFun.jl`.
2. Mark each export as one of: already documented, needs example, needs background, or needs dedicated page.
3. Record any public helpers that should remain internal and should not be promoted into the docs.
4. Confirm the docs build still succeeds before adding content.

Exit criteria:

- Every exported function has a known documentation status.
- The doc build is green before the content pass begins.

## Phase 2: API reference pages

1. Add or expand per-function docstrings for the public Faddeeva routines.
2. Add or expand per-function docstrings for the AMOS routines.
3. For each public function, include a short purpose statement, argument summary, return-value summary, and notable edge cases.
4. Keep the existing autogenerated module index, but make the per-function entries self-contained enough to serve as reference material.

Exit criteria:

- Each exported symbol has at least one meaningful docstring entry surfaced by `@autodocs`.
- The API reference is readable without opening the source file.

## Phase 3: Examples

1. Add minimal usage examples for the main entry points in `docs/src/index.md` or dedicated example pages.
2. Include at least one normal-case example and one edge-case example for functions with special branch behavior.
3. Prefer examples that are small enough to remain stable across platforms.
4. Avoid benchmarking examples inside the primary narrative unless they directly illustrate the API.

Exit criteria:

- The homepage shows how to call the main functions.
- The documented examples match the current exported signatures.

## Phase 4: Provenance and math notes

1. Link the public API to the original OpenSpecFun source files using the provenance map already present in the module.
2. Add concise notes on the mathematical definition or identity behind each family of functions where that helps the reader.
3. Cite relevant literature only where it clarifies behavior or implementation choices.
4. Keep the focus on readable reference notes, not on reproducing full derivations.

Exit criteria:

- Readers can trace each public function back to its upstream source family.
- The docs explain the mathematical intent of the implementation.

## Phase 5: Plots and benchmarks

1. Decide which functions benefit from visual plots and which are better left as text-only reference.
2. Add only lightweight, reproducible plots that can run in the docs build environment.
3. Add benchmarks only for functions where a simple comparison adds value and does not create fragile docs output.
4. Keep plots and benchmarks out of the critical path if they slow the build or make it unstable.

Exit criteria:

- Any added plot or benchmark is deterministic and documented.
- The docs build remains fast enough to be practical.

## Phase 6: Final verification and release

1. Build the docs locally.
2. Inspect the rendered output in a browser.
3. Fix any broken links, missing exports, or confusing API descriptions.
4. Commit the documentation work and push only to `origin`.

Exit criteria:

- The rendered documentation is readable in a browser.
- Changes are committed and pushed to the fork remote, not upstream.
